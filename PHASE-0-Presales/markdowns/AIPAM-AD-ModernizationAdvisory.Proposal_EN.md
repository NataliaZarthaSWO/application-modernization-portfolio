Well Architected Solution Framework
To carry out the review, we will rely on the Well Architected Framework practice, which allows us to understand the current and desired state of an application across five different pillars:
Reliability: The ability of a system to recover from infrastructure or service failures, such as dynamically acquiring computing resources to meet demand, mitigating potential service disruptions due to misconfigurations or temporary network issues, and releasing resources when no longer needed.
Security: The ability to protect systems and information assets while delivering all business services supported by the application. This involves risk assessments and mitigation strategies.
Operational Excellence: The ability to run and monitor the application to take preventive actions and continuously improve support processes and procedures.
Performance Efficiency: The ability to use computing resources efficiently to meet system requirements and maintain that efficiency as demand changes and technologies evolve.
Cost Optimization: The ability to avoid or eliminate unnecessary costs from underutilized resources.
Once the analysis is completed, a chart called "Gap Analysis" is generated, showing the current and desired maturity level of the application for each pillar. This will help define the best strategies and technologies to close the gaps.
It will also help identify the most appropriate modernization alternative to achieve the goals of the modernization strategy.
Optimize: This approach is based on Platform as a Service (PaaS) and enables additional cost savings in platform management, while also providing service level agreements (SLAs) for platform operations. It may involve some changes to the application to ensure compatibility with the PaaS environment.
Transform: This approach leverages cloud-native technologies aligned with business demands, offering all the necessary tools to grow in response to market challenges. It requires new architecture to take full advantage of the target technologies.

Application Source Code Analysis and Rationalization Matrix
During the application rationalization process, we classify applications into one of four quadrants (Tolerate, Optimize, Transform, Retire).
The Classification matrix is a graph in two dimensions where the horizontal axis illustrates the technical condition which is calculated using the Software Health metric, and the vertical axis illustrates the importance of the application to the business (for example the competitive advantages that it creates for the organization).
After finishing the uploading of the information of each application into the analysis tool, the applications will be classified into one of four quadrants:
Quadrant 1. (Transform). Applications that have a high value for the business, but the software health is low. This prevents the business from growing at the speed that is needed. For these applications, a transformation process (Refactor, Rebuild) should be carried out, seeking to increase the software’s health and improving response times to business needs.
Quadrant 2. (Optimize). Applications that have a high value for the business, and their software health is high. The applications located in this quadrant are candidates to be improved by taking them to PaaS following a Replatform strategy and are candidates to add innovative features with modern technologies to generate competitive advantage in the market.
Quadrant 3. (Shut Off) Applications that have low software health and a low value for the business; these applications are candidates to be Retired, which means running a process to extract some functionalities and to shut down other functional parts.
Quadrant 4. (Tolerate). Applications with high software health but with low value for the business. No action will be performed on these applications beyond the typical maintenance and evolution, that is, they will be tolerated.  The apps need to be kept as-is in the current environment.
To support this exercise, we leverage AI-based tools that enable rapid inspection of each application's source code to determine its health status, as well as an easy way for stakeholders to respond to surveys that help assess business impact.

AI-Powered Strategic Analysis
In this component, the Application Modernization Consultant performs a strategic analysis to identify the most suitable modernization path for each application, considering four modernization options (Rehost, Replatform, Refactor, and Rebuild) for both the Application and Database layers. To support this process, AI agents are used to consolidate technical and business inputs, analyze scenarios, and inform decision-making. Based on this analysis, the consultant defines the optimal modernization path to achieve the stated objectives.The available alternatives are:
Rehost: Also known as “lift and shift,” this approach consists of moving an application as-is to a cloud environment to take advantage of IaaS (Infrastructure as a Service) benefits. The objective is to transition from on-premises infrastructure to the cloud by running the workload on cloud-based virtual machines with similar capabilities, without modifying the application’s code or functionality.
Replatform: This approach involves migrating an application to the cloud while making a minimal set of non-disruptive changes to the code, without altering its core structure, features, or functionality. The objective is to enable improved resilience, scalability, security, and cost optimization by leveraging PaaS (Platform as a Service) capabilities.
Refactor: This option involves restructuring and optimizing the existing codebase, without changing its external behavior, in order to reduce technical debt and improve non-functional attributes. The objective is to fully leverage cloud-native services and achieve a higher level of cost and performance optimization.
Rebuild: This approach consists of redesigning and rewriting the application component from the ground up, while preserving its original scope and specifications, using a cloud-native architecture and technologies.

SoftwareONE will propose a set of DevOps practices suited for the selected modernization strategy, with the goal of using  quality standards, providing agility in the development process and improving time to market.
DevOps is a framework that proposes a collaborative working relationship between the development team and the operations team that increases reliability, stability, resilience, and security of the production environment.
DevOps incorporates several agile principles, methods, and practices such as collaboration, automation, continuous delivery, continuous integration, continuous testing, continuous deployment, continuous monitoring, and rapid recovery.
The following graph presents the main DevOps practices that, in our experience, can be implemented:
Url interna: DevOps Framework
Development team: This group of practices is in charge of defining the practices that are developed during the continuous (daily) execution of tasks by the development team. The development team includes functional analysts, development engineers, test analysts, UX analysts, among other roles involved in building the solution.
Artifact repository:
This set of practices deals with the management of artifacts that are generated throughout the software development cycle, although they are part of the development team they are part of cross-cutting practices that seek to ensure artifacts maintain their control, availability, and traceability.
Continuous Integration:  This set of practices is responsible for ensuring that the source code is stable both in its integrity and in its verification, to generate the artifacts that will be used in the deployment of the solution in the different environments.
Continuous Deployment: These practices refer to the installation in test environments or the production environment where the solution will be available for use.  In the DevOps Framework these practices are contemplated at two different times because in test environments, the set of “Continuous Testing” practices must be considered.
Continuous Testing:
This set of practices refers to the verifications that must be carried out in a different environment from that of development where there is access to services and testing environments or UAT similar to the production environment and therefore verifications can be made to the solution in addition to those performed in the code quality analysis and the execution of unit tests.
Continuous Operation:
After deployment in productive environments, mechanisms must be activated so that the operation of the solution is not impacted by internal or external elements, which ensures availability that in turn generates continuous value for the customer. As part of the development, the creation of functionalities or configuration of tools that allow the monitoring of application health and minimize the time required to perform analysis when incidents occur, This includes adequate registration of exceptions, execution time measurements of the functionalities of the application and the services on which it depends, and the dashboards to monitor the state of the systems on which it is dependent.
